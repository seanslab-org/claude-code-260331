Where Memory plugs into Docknote

Three seams · four decisions · 2026-05-05
Companion to docknote-architecture.html (as-built) and docknote-brainmemory-design.md (Phase 3 spec).

Memory doesn’t slot into one layer of Docknote — it threads through three. But only one of those three is architecturally novel; the other two follow the existing pattern. Below: the overlay diagram, a short tour of each seam, and the four decisions hidden inside seam A that are actually load-bearing.

2.What each seam adds

SEAM A

Storage — the only architecturally novel seam

Today, Docknote has no global index over content. Per-meeting index.md + a flat catalog.jsonl roll-up is the entire cross-meeting story. Memory introduces:

  • ~/Docknote/.brain/engrams.sqlite — sentence vectors, RAPTOR cluster IDs, entity-name resolution table. Indexes only, per principle 1. First time SQLite enters the architecture for content.
  • ~/Docknote/meetings/<id>/engrams.md — sentence text + stable IDs, born in the meeting folder like index.md is. Co-located with truth so vectors can be re-derived from markdown if SQLite ever corrupts.

This is the load-bearing decision. Everything else follows from it. Four sub-decisions live inside this seam — see §3.

SEAM B

Pipeline tail — follows existing pattern

batch_ingest.py today ends at aggregate_index.py. Memory adds two stages, both following the existing pattern:

  • extract_engrams.py — sentence-segment, embed (local CoreML), write to engrams.md + SQLite. Runs per meeting, synchronous tail of the import pipeline. Wrapped with snapgpu_marker so the existing “Summarizing” chip covers it — no new pipeline UI stage.
  • consolidate_engrams.py — RAPTOR clustering pass over the whole engram corpus + entity merge/split. Runs occasionally (cron-ish, every N new engrams), uses SnapGPU for cluster-summary LLM calls. Genuinely new because nothing today runs corpus-wide.
SEAM C

UI + IPC — three new surfaces

  • MemoryView (Phase-3 placeholder today) → galaxy + vertical time column + floating ask + people lens. Behind commands/memory.rs. Locked mockup: rachel-redesign/v4.3-memory-galaxy-floating-ask.html.
  • Brain Inbox — Geomi confirmation queue. Either its own sidebar item or a modal launched from a Memory badge. Behind commands/geomi.rs. This is the surface where principle 2 lives — every consequential write routes through here.
  • Ask integration — sibling commands/ask_memory.rs rather than overloading ask_meeting. Prompts and retrieval shape diverge enough (scope×person, time×person) that one module would get messy.

3.Four decisions hidden in seam A

These are what’s actually unsettled. Each is a fork; each fork affects how the rest looks. Recommendations marked ★.

1

Engram granularity

A 90-min meeting has 200+ sentences. What’s a galaxy dot?

A

Sentence-level dot — every sentence is an engram. Dense galaxy, expensive consolidation, hard to scan.

★ B

Passage-level dot — coherent passages (~10–20 per meeting). The mockup’s “one dot per meeting” only works if engram = passage. Sentence vectors still indexed underneath for retrieval, but the dot is the passage.

2

Embedding compute

Where does the embedding model run?

★ A

Local CoreML — sentence-transformers exported to CoreML, runs on the user’s Mac. Better for privacy, latency, offline. Same shape as the pyannote-on-CoreML pattern already in the pipeline.

B

Remote on SnapGPU — keeps one model in one place. Adds a network hop per ingest, breaks offline.

3

Consolidation cadence

When does RAPTOR re-cluster?

★ A

Per-import (sync) for the first ~1k engrams. Simple, predictable, no background-job state to manage. Ingest stays fast at small corpora.

★ B

Background pass after ~1k engrams. Triggered by import count delta. Keeps ingest snappy; introduces background-job state. Both modes coexist; the threshold is a config knob.

4

Entity resolution timing

When does “Alex” become person_id?

★ A

Eager (at ingest) — resolve during extract_engrams, write person_id into the entity table. Makes the killer-move (scope × person) a cheap join. Mistakes get fixed via Geomi confirmation.

B

Lazy (at query) — keep raw names, resolve when the user pins someone. Fast ingest, slow query, and the killer-move becomes a multi-step lookup.

4.What does NOT change

Memory is additive. None of the existing system changes shape.
  • The markdown corpus stays untouched — meetings/<id>/{meta, transcript, summary, index, todos, audio} is unchanged.
  • The pipeline_queue single-flight FIFO continues to drain one import at a time; engram extraction is just another stage in the existing detached child.
  • Notes / Whispers / Todos / HiDock / Settings — no Memory wiring.
  • Tauri commands, Keychain, user.json, backup zip format — unchanged.
  • SnapGPU consumed exactly as today (OpenAI-compatible HTTP); only the prompts in consolidate_engrams are new.
Drawn 2026-05-05 in service of the “memory itself is still not 100% sure” conversation. The four decisions in §3 are the next gate; once those are picked, the storage schema (table list, column types, index keys) follows mechanically.
